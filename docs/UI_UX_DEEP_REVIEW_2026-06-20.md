# Recap UI/UX Deep Review And Consensus Proposal

작성일: 2026-06-20
검토 방식: UI 디자이너, UX 디자이너, 기능기획자 독립 리뷰 후 합의
대상: Recap 회의 전사 SPA, Decision Wiki, 검색/채팅/설정, 첫 실행 경험
검증 범위: 코드/문서 정적 리뷰, 임시 데이터 디렉터리 기반 로컬 서버 화면 확인

## 1. 만장일치 결론

Recap은 "로컬 AI 파이프라인 콘솔"이 아니라 "Apple Silicon Mac에서 민감한 한국어 회의를 로컬로 처리하고, 근거 인용 가능한 Decision Wiki로 축적하는 회의 기억 앱"으로 정렬해야 한다.

현재 UI는 macOS 네이티브 톤, 접근성 기초, 명령 팔레트, STT 수동 복구, 업로드, 검색, 채팅, Wiki까지 기능 자산이 풍부하다. 그러나 사용자가 처음 만나는 제품 표면은 회의 사용자보다 기술 운영자에게 더 친절하다. 홈은 "첫 회의 만들기"보다 통계와 "회의를 선택하세요"를 먼저 말하고, 검색은 벡터/FTS/점수/SPEAKER_00 같은 내부 용어를 노출하며, 설정은 LLM/STT/환각 필터/백필/A-B 테스트가 거의 같은 무게로 배치된다.

베타 릴리즈 기준은 단순히 "전사가 된다"가 아니다. 첫 성공 회의 후 사용자가 바로 다음 산출물을 확인할 수 있어야 한다.

- 전사문
- 요약
- 결정사항
- 액션 아이템
- 프로젝트 현황
- 원문 citation과 timestamp 이동
- 검색/채팅 준비 상태

## 2. 근거 요약

제품 철학은 이미 분명하다. `docs/design.md`는 "Apple이 만들었다고 착각할 정도로 자연스러운 macOS 네이티브 웹앱"을 목표로 삼고, Familiarity, Restraint, Responsiveness를 핵심 원칙으로 둔다. `docs/plans/2026-05-21-decision-wiki-product-plan.md`는 제품 가치가 "AI와 채팅하기"가 아니라 "회의에서 나온 의사결정이 사라지지 않도록 구조화하고, 언제/왜/누가 결정했는지 원문 근거와 함께 찾게 하는 것"이라고 정의한다.

현재 구현은 이 방향으로 가고 있지만 표면의 위계가 흔들린다.

- 홈: 통계 4개와 빠른 작업 뒤에 "회의를 선택하세요"가 표시된다. 회의가 0건일 때 본문 CTA는 첫 회의 생성보다 선택을 요구한다.
- 사이드바: 빈 목록에는 "녹음 시작" CTA가 있으나 실패 시 콘솔에만 기록되는 경로가 있다.
- 가져오기: 업로드 API와 JS 업로드 흐름이 존재하지만, HTML에는 "업로드 API는 아직 준비 중" 문구가 남아 있다.
- 검색: 기본 설명과 결과 통계에 "벡터", "FTS", "점수"가 드러난다.
- 채팅: 회의가 없는 상태에서도 입력이 활성화되고 추천 질문이 표시된다.
- 위키: "Decision Wiki"가 핵심 가치인데 `/app/wiki`는 현황/검색 중심 일반 Wiki처럼 읽힌다.
- 설정: 일반 탭에 LLM 모델, 파이프라인, 자동 처리, 환각 필터, 저장소 관리, STT 모델, A/B 테스트가 모두 들어간다.
- 모바일/좁은 창: 390px 폭에서 하단 탭바와 본문 빈 상태가 충돌하고, drawer open 상태에서 전역 테마 버튼이 시각 위계상 남아 있다.
- 디자인 시스템: `docs/design.md`의 spacing/radius 토큰과 `ui/web/tokens.css`의 실제 토큰 값이 일부 다르다.

## 3. 3인 끝장토론

### 쟁점 A: 이 제품의 주인공은 무엇인가?

UI 디자이너 의견:
기본 UI는 "회의 기억 앱"처럼 보여야 한다. 지금은 통계, 리소스, 모델, 인덱스, 백필이 같은 레벨에 보여서 제품이 기술 콘솔처럼 읽힌다.

UX 디자이너 의견:
첫 사용자가 원하는 것은 "첫 회의가 성공적으로 전사되고 나중에 찾을 수 있는가"다. 설치와 모델/토큰/오디오 실패 가능성이 크므로 앱 안에 준비 체크가 필요하다.

기능기획자 의견:
장기 차별점은 단순 전사보다 Decision Wiki다. 민감한 회의를 로컬로 처리하고, 원문 근거가 있는 의사결정 DB를 만드는 것이 가장 강한 제품 계약이다.

합의:
제품 포지션은 "로컬 우선 회의 기억 앱"이고, 핵심 산출물은 Decision Wiki다. 전사와 채팅은 독립 목적이 아니라 Decision Wiki와 검색 가능한 회의 기억을 만들기 위한 흐름이다.

### 쟁점 B: 첫 화면은 무엇이어야 하는가?

UI 디자이너 의견:
회의가 0건이면 통계보다 녹음/가져오기가 앞서야 한다. 현재 본문 빈 상태는 "회의를 선택하세요"라고 말해 첫 사용 상황과 맞지 않는다.

UX 디자이너 의견:
첫 화면은 readiness check가 되어야 한다. HF 토큰, STT 모델, 오디오 입력, 저장 폴더, 샘플 녹음/업로드 테스트가 통과되어야 첫 회의 실패를 줄일 수 있다.

기능기획자 의견:
데이터가 있는 사용자의 홈은 Wiki 현황이 되어야 한다. 결정, 액션, 프로젝트 현황이 바로 보이는 것이 제품 약속과 맞다.

합의:
홈은 상태 적응형으로 바꾼다.

- 첫 실행: 준비 체크리스트
- 회의 0건: "첫 회의 만들기" 중심 CTA
- 회의 있음: Decision Wiki 현황과 최근 회의
- 처리 중: 산출물 기준 진행 상태
- 문제 있음: 영속적인 복구 카드

### 쟁점 C: 검색, 채팅, Wiki는 어떻게 나눌 것인가?

UI 디자이너 의견:
검색, 채팅, 위키가 모두 같은 무게로 있으면 사용자는 "지난 결정"을 어디서 찾아야 할지 모른다.

UX 디자이너 의견:
채팅은 데이터가 없거나 인덱스가 준비되지 않았을 때 입력을 열어두면 안 된다. 근거 없는 답변은 로컬 회의 기반 제품의 신뢰를 해친다.

기능기획자 의견:
Decision Wiki 계획 문서의 결론처럼 "search before chat"이 맞다. 채팅은 주 인터페이스가 아니라 검색과 결정 상세의 보조 인터페이스다.

합의:
역할을 명확히 분리한다.

- 회의: 원문, 요약, 재생, 편집
- 결정사항: 결정/액션/프로젝트/사람을 보는 1급 업무 화면
- 검색: 원문과 결정 기록을 찾는 통합 검색, 내부 알고리즘 세부 정보는 상세 모드
- 채팅: 검색 결과와 citation을 바탕으로 질문에 답하는 보조 인터페이스

### 쟁점 D: 설정은 얼마나 보여줄 것인가?

UI 디자이너 의견:
일반 설정이 너무 무겁다. macOS 시스템 설정 같은 카드 스타일은 좋지만, 기본 사용자에게 위험한 값이 너무 많이 노출된다.

UX 디자이너 의견:
STT 수동 다운로드와 복구 흐름은 훌륭하지만 첫 화면에는 "추천 모델 사용"과 "문제 해결"만 있으면 된다.

기능기획자 의견:
A/B 테스트, 커스텀 모델, 라우터, 백필은 핵심 루프 검증 뒤 확장해야 한다. 베타 전에는 첫 성공 회의와 citation 신뢰가 우선이다.

합의:
설정은 "추천 기본값 + 고급 설정"으로 재편한다.

- 기본: 모델 자동/추천, 저장 위치, 오디오 입력, 테마, 언어
- 문제 해결: STT 수동 다운로드, 인덱스 복구, Wiki 백필
- 고급: LLM backend, temperature, 환각 필터, A/B 테스트, 커스텀 모델

### 쟁점 E: 기술 투명성은 어디까지 필요한가?

UI 디자이너 의견:
벡터, FTS, RAG, CER, WER, score는 기본 UI에서 제거해야 한다.

UX 디자이너 의견:
사용자는 정확도와 신뢰를 알아야 하지만 알고리즘 이름보다 "출처 있음", "원문으로 이동", "검색 준비 완료"가 더 중요하다.

기능기획자 의견:
민감 데이터 로컬 처리라는 차별점은 명확히 말해야 한다. 다만 "완전 무인터넷"처럼 오해될 표현은 피해야 한다. 모델 다운로드는 필요하고, 회의 데이터는 외부 API로 보내지 않는다는 계약이 정확하다.

합의:
기본 UI는 업무 언어를 사용한다. 기술 세부는 "세부 정보", "진단", "고급"에 넣는다.

## 4. 화면별 개선 제안

### 홈

현재:
회의가 없어도 대시보드 통계가 먼저 나오고, 본문은 "회의를 선택하세요"라고 말한다.

개선:

- 첫 실행에서는 "준비 체크리스트"를 표시한다.
- 회의가 0건이면 "녹음 시작"과 "오디오 가져오기"를 본문 중앙 1차 CTA로 둔다.
- 통계는 회의가 생긴 뒤 표시한다.
- "전체 일괄", "최근 24시간"은 회의가 없을 때 숨긴다.
- 홈의 핵심 KPI는 "오늘의 결정", "열린 액션", "최근 회의", "검색 준비 상태"로 바꾼다.

### 오디오 가져오기

현재:
모달은 안정적이지만 "파이프라인 시작" 버튼과 "업로드 API 준비 중" 문구가 실제 구현과 어긋난다.

개선:

- 버튼 문구: "업로드하고 처리 대기열에 추가"
- 성공 문구: "업로드 완료. 감지 후 전사가 시작됩니다."
- 실패 문구: 파일별 실패 원인과 재시도 버튼 표시
- 2GB 제한과 지원 포맷은 유지하되, 모달 안에서 실제 저장 폴더는 접을 수 있는 세부 정보로 내린다.

### 회의 상세

현재:
전사/요약/검색/재생의 기반은 좋다. 다만 많은 액션이 같은 레벨로 노출될 위험이 있다.

개선:

- 상단 1차 액션: "요약 생성" 또는 "검색 준비 완료 확인"처럼 현재 상태의 다음 행동만 노출
- 보조 액션: 복사, 다운로드, 전체 바꾸기
- 위험 액션: 재전사, 삭제, 취소는 overflow와 확인 단계로 이동
- 실패 시 토스트보다 회의 상세 상단에 복구 카드 표시
- 진행 단계는 `녹음됨 → 전사 중 → 화자 구분 중 → 요약 중 → 검색 준비 중 → Decision Wiki 반영 중 → 완료`처럼 산출물 기준으로 표시

### 검색

현재:
검색 초입은 좋지만 내부 용어가 기본 화면에 노출된다.

개선:

- 기본 결과 통계: "관련 결과 12건"으로 단순화
- 상세 토글: "검색 세부 정보"에서 벡터/키워드/점수 표시
- 화자 필터: `SPEAKER_00` 대신 사용자가 부여한 이름 또는 "화자 1" 표시
- 검색 대상 토글: "전체", "전사문", "결정사항", "액션"
- 회의가 0건이면 검색 추천어 대신 녹음/가져오기 CTA 표시

### 채팅

현재:
회의가 없어도 추천 질문과 입력창이 활성화된다.

개선:

- 인덱스가 없으면 입력창을 비활성화하고 "회의를 먼저 추가하세요" CTA 제공
- 출처 없는 답변은 기본적으로 금지한다.
- 답변에는 사용한 회의/결정/citation을 항상 붙인다.
- "전체 회의"와 "현재 회의" 범위를 명확히 분리한다.
- 채팅은 검색 결과를 해석하는 보조 화면으로 포지셔닝한다.

### Decision Wiki

현재:
일반 Wiki 현황/검색 화면처럼 보이고, 결정사항이 1급 객체로 충분히 드러나지 않는다.

개선:

- 기본 탭을 "결정사항" 또는 "현황"으로 두되 카드 구조를 결정 중심으로 재설계
- 1급 화면:
  - 결정 목록
  - 결정 상세
  - 액션 아이템
  - 프로젝트 현황
  - citation 검증 상태
- 빈 상태는 "회의를 처리하면..." 설명만 두지 말고 "샘플 회의 가져오기", "기존 회의 백필", "첫 회의 녹음" 액션 제공
- Wiki health는 우측 작은 배지가 아니라 "근거 검증 상태"와 "백필 필요"로 업무 언어화

### 설정

현재:
일반 탭이 제품 운영 콘솔처럼 길다.

개선:

- 기본 탭: 일반 사용자 설정만 표시
- 모델 탭: 추천 모델 카드 1개와 "다른 모델 보기" progressive disclosure
- 문제 해결 탭: 수동 다운로드, 인덱스 복구, Wiki 백필
- 고급 탭: LLM backend, temperature, A/B 테스트, 환각 필터
- 저장 방식은 가능하면 즉시 저장과 인라인 status로 통일한다. 위험한 설정은 명시 저장을 유지한다.

### 모바일/좁은 창

현재:
하단 탭바, 햄버거, drawer가 존재하지만 본문 빈 상태가 아래로 밀리고 전역 버튼이 drawer 위계와 충돌한다.

개선:

- 이 제품의 우선순위는 "모바일 웹"보다 "작은 macOS 창"으로 정의한다.
- 768px 이하에서 통계 카드보다 첫 CTA를 먼저 보여준다.
- drawer open 상태에서는 전역 테마/홈/리소스 버튼을 시각적으로도 비활성화한다.
- 하단 탭바와 main padding-bottom을 더 명확히 맞춘다.

## 5. 우선순위 로드맵

### P0: 베타 전 필수

1. 제품 계약 정리
   - 문서, 설정, UI 카피를 "로컬 우선 Decision Wiki"로 맞춘다.
   - 로컬/프라이버시 문구는 "회의 데이터는 외부 API로 보내지 않음, 초기 모델 다운로드는 필요"로 정확히 쓴다.

2. 첫 실행 준비 체크리스트
   - Python/ffmpeg 같은 개발 환경은 앱 내부보다 설치 과정에서 확인하되, 앱에서는 다음 상태를 보여준다.
   - HF token/pyannote 접근 가능
   - STT 모델 준비
   - LLM 모델 준비
   - 오디오 입력 장치
   - 저장 폴더 권한
   - 샘플 업로드/녹음 테스트

3. 홈 재설계
   - 회의 0건: 첫 회의 생성
   - 처리 중: 산출물 기준 진행
   - 회의 있음: Decision Wiki 현황

4. 기술 용어 감추기
   - 기본 UI에서 RAG, FTS, vector, score, CER/WER, SPEAKER_00를 숨긴다.
   - 고급/진단 모드에서만 세부 노출한다.

5. 검색/채팅 신뢰 가드
   - 회의/인덱스가 없으면 검색 추천과 채팅 입력을 제한한다.
   - 출처 없는 답변을 금지하거나 명확히 "회의 근거 없음"으로 표시한다.

6. Wiki를 1급 제품 표면으로 승격
   - 결정 목록/상세/citation을 중심으로 `/app/wiki`를 재편한다.
   - Wiki disabled/dry-run 상태라면 명확한 활성화/백필 CTA를 제공한다.

7. 실패 복구 카드
   - 토스트만으로 실패를 처리하지 않는다.
   - 회의 상세와 홈 건강 상태에 실패 원인, 영향, 추천 액션, 재시도 버튼을 남긴다.

8. 카피 drift 수정
   - 업로드 모달의 "API 준비 중" 문구 제거
   - README, AGENTS, config, UI 기본 모델명/설명 정합성 점검

## 6. 설득 프로세스

이 제안은 다음 순서로 팀을 설득하는 것이 가장 현실적이다.

1. 문제를 "예쁘게 만들기"가 아니라 "첫 성공 회의와 신뢰 가능한 결정 기록"으로 정의한다.
2. 첫 사용자에게 현재 실패 가능한 지점을 보여준다: 토큰, 모델, 오디오, 업로드, 인덱스, Wiki.
3. 기술 세부를 숨기는 것이 정보 은폐가 아니라 기본 업무 성공률을 높이는 일임을 설명한다.
4. 고급 기능을 삭제하지 않고 progressive disclosure로 이동한다고 명확히 한다.
5. 베타 성공 지표를 합의한다.

권장 베타 성공 지표:

- 첫 회의 완료율
- 첫 회의 완료까지 걸린 시간
- 업로드/녹음 실패 후 복구율
- 검색 준비 완료까지 걸린 시간
- 결정사항 자동 추출 precision
- citation 클릭 성공률
- 출처 없는 채팅 답변 비율
- 설정 화면에서 저장 없이 이탈하는 비율

## 7. 최종 합의문

세 관점은 다음 문장에 만장일치로 동의한다.

Recap v0.1 beta는 범용 회의 전사기나 AI 챗봇이 아니라, Apple Silicon Mac에서 한국어/민감 회의를 로컬로 처리해 근거 인용 가능한 Decision Wiki를 만드는 제품이다. 기본 화면에서는 녹음/가져오기, 회의, 전사문, 요약, 결정사항, 액션, 검색, 출처만 남기고, 모델·인덱스·백필·A/B·리소스 상태는 고급/진단 영역으로 분리한다. 릴리즈 기준은 첫 성공 회의 후 사용자가 결정, 액션, 프로젝트 현황, 원문 citation을 바로 확인할 수 있는가이다.
