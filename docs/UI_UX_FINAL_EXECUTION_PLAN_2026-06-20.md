# Recap UI/UX Critical Revalidation And Final Execution Plan

작성일: 2026-06-20
상태: 최종 실행 계획
대상: Recap SPA, 첫 실행 경험, 검색/채팅, Decision Wiki, 설정, 모바일/좁은 창

## 1. 재검증 결론

기존 `UI_UX_DEEP_REVIEW_2026-06-20.md`의 큰 방향은 타당하다.

Recap은 단순 전사기나 AI 채팅창이 아니라, Apple Silicon Mac에서 한국어/민감 회의를 로컬로 처리하고 근거 인용 가능한 Decision Wiki로 축적하는 회의 기억 앱으로 정렬해야 한다.

다만 최종 계획에서는 일부 표현을 더 엄밀하게 수정한다.

1. "Wiki가 꺼져 있다"는 표현은 반만 맞다.
   `config.py`의 `WikiConfig.enabled=False`, `dry_run=True`, `router_enabled=False` 기본값은 맞다. 그러나 `config.yaml`에는 `wiki.ranking`, `wiki.semantic`, `wiki.digest`가 켜져 있고 `/app/wiki` 화면도 존재한다. 따라서 문제는 "Wiki 전체 부재"가 아니라 "Decision Wiki 컴파일/라우터/제품 표면이 기본 성공 흐름으로 묶이지 않는다"가 정확하다.

2. "업로드 API가 없다"는 표현은 틀렸다.
   실제 `/api/uploads` 호출이 구현되어 있다. 문제는 `index.html`과 `list-panel.js`에 남은 "업로드 API 준비 중" 문구와 주석, 그리고 버튼명 "파이프라인 시작"이 현재 동작과 맞지 않는 카피 drift다.

3. 모바일 충돌은 P0로 단정하지 않는다.
   390px Playwright 스냅샷에서 하단 내비게이션, drawer, 홈 화면 구조는 존재한다. 다만 작은 macOS 창/모바일에서 정보 우선순위가 나쁘고 추가 시각 QA가 필요하다. 따라서 모바일은 베타 차단 P0가 아니라 P1/P2 품질 항목으로 둔다.

4. 설정 화면은 "모든 고급 기능이 일반 탭에 있다"보다 더 구체적으로 본다.
   A/B 테스트는 이미 고급 섹션에 있으나, LLM backend, temperature, LLM 스킵, 자동 처리, 환각 필터, 저장소, STT 모델 카드가 일반 탭에 함께 있어 첫 사용자에게 운영 콘솔처럼 보인다.

5. 채팅의 신뢰 리스크는 확인됐다.
   검색 결과가 없을 때 백엔드는 "알고 있는 범위에서 답변"하도록 프롬프트를 만든다. 이것은 "회의 근거 기반" 제품 약속과 충돌한다. 채팅은 출처 없는 답변을 기본 금지하거나 명확히 "회의 근거 없음"으로 분리해야 한다.

## 2. 검증 근거

검증 방식:

- 저장소 상태 확인: 기존 미추적 문서 2개를 보존했다.
- 정적 코드 검토: `config.py`, `config.yaml`, `ui/web/*`, `search/chat.py`, `core/pipeline.py`, `docs/design.md`, Decision Wiki 계획 문서 확인.
- 실제 렌더링 검토: 임시 `MT_BASE_DIR`로 `127.0.0.1:9877` 서버를 띄우고 `/app`, `/app/search`, `/app/chat`, `/app/wiki`, `/app/settings`, import modal, 390px viewport를 Playwright 스냅샷으로 확인했다.
- 제외: 실제 STT/LLM 모델 처리, 실제 회의 golden set, Lighthouse 점수, 전체 E2E 테스트는 실행하지 않았다.

확인된 사실:

- 홈 첫 화면은 회의 0건에서도 통계 4개, 빠른 작업, "전체 일괄", "최근 24시간", "회의를 선택하세요"를 보여준다.
- 왼쪽 빈 회의 목록은 "녹음 시작" CTA를 제공하지만 실패 처리가 콘솔 로그에 머문다.
- import modal은 파일 선택 UI와 `/api/uploads` 업로드 흐름이 있으나, "업로드 API는 아직 준비 중" 문구와 "파이프라인 시작" 버튼명이 남아 있다.
- 검색 화면은 기본 안내와 결과 통계에 벡터/FTS/점수/SPEAKER_00 같은 내부 용어를 노출한다.
- 채팅 화면은 회의가 없어도 추천 질문과 입력창을 보여준다.
- 채팅 백엔드는 검색 컨텍스트가 없을 때도 일반 지식 답변으로 넘어갈 수 있다.
- Wiki 화면은 존재하지만 "결정사항 목록/상세/citation" 중심의 1급 업무 화면보다는 현황/검색 Wiki 화면에 가깝다.
- 설정 일반 탭은 LLM/STT/파이프라인/자동 처리/환각 필터/저장소가 한 화면에 모여 있다.
- `docs/design.md`의 spacing/radius 토큰과 `ui/web/tokens.css`의 실제 토큰은 일부 불일치한다.
- 파이프라인 실제 단계는 `CONVERT → TRANSCRIBE → DIARIZE → MERGE → CORRECT → SUMMARIZE → CHUNK → EMBED`이고 Wiki compile은 별도 후속 단계인데, 뷰어 진행 UI는 6단계 중심이다.

## 3. 최종 제품 원칙

1. 첫 성공 회의가 최우선이다.
   사용자는 모델이나 인덱스를 관리하러 온 것이 아니라 첫 회의를 안전하게 기록하고 다시 찾으러 온다.

2. Decision Wiki가 제품의 중심이다.
   전사문, 요약, 검색, 채팅은 독립된 기능이 아니라 결정사항, 액션, 프로젝트 현황, 원문 citation을 만드는 경로다.

3. 기본 UI는 업무 언어를 쓴다.
   RAG, FTS, vector, score, CER/WER, backend, dry-run 같은 용어는 고급/진단 영역으로 내린다.

4. 신뢰는 citation으로 만든다.
   결정사항과 채팅 답변은 원문 회의, timestamp, speaker 근거로 돌아갈 수 있어야 한다.

5. 고급 기능은 삭제하지 않고 접는다.
   STT 모델, LLM backend, A/B 테스트, 백필, 라우터, 환각 필터는 중요하다. 다만 첫 사용자 기본 경로에서는 드러내지 않는다.

## 5. P0: 베타 전 필수 계획

### P0-1. 첫 실행 준비 체크리스트

목표:
첫 사용자가 "왜 실패했는지"를 터미널이나 문서 없이 앱에서 이해하게 한다.

범위:

- HF token/pyannote 접근 상태
- STT 모델 준비 상태
- LLM 모델 준비 상태
- 오디오 입력/녹음 가능 상태
- 저장 폴더 권한
- 샘플 업로드 또는 짧은 녹음 테스트
- 검색 인덱스/Wiki 생성 가능 상태

완료 조건:

- 회의 0건 첫 실행에서 "회의를 선택하세요"가 아니라 준비 체크리스트가 보인다.
- 각 항목은 정상/주의/필수 조치 상태와 다음 액션을 가진다.
- HF token이 없으면 모델 약관/토큰 설정 안내로 연결하고, 토큰 우회나 SSL 우회 흐름은 제공하지 않는다.

### P0-2. 홈 화면 상태 적응형 재설계

목표:
홈을 "대시보드 숫자"가 아니라 다음 행동을 알려주는 시작점으로 만든다.

상태별 화면:

- 첫 실행: 준비 체크리스트
- 회의 0건: 녹음 시작, 오디오 가져오기
- 처리 중: 산출물 기준 진행 상태
- 회의 있음: 최근 회의, 최근 결정, 열린 액션
- 문제 있음: 복구 카드

완료 조건:

- 회의 0건에서 통계 카드보다 첫 회의 CTA가 먼저 보인다.
- "전체 일괄", "최근 24시간"은 회의가 없을 때 숨긴다.
- 글로벌 리소스 바는 기본 홈에서 시각 우선순위를 낮춘다.

### P0-3. 업로드/가져오기 카피 drift 수정

목표:
이미 구현된 업로드 흐름과 UI 문구를 맞춘다.

변경:

- "업로드 API는 아직 준비 중입니다" 제거
- "파이프라인 시작" → "업로드하고 처리 대기열에 추가"
- 성공 메시지: "업로드 완료. 전사 대기열에 추가되었습니다."
- 실패 메시지: 파일별 실패 원인과 재시도 행동 표시
- 오래된 JS 주석도 실제 `/api/uploads` 동작 기준으로 갱신

완료 조건:

- import modal에서 사용자가 직접 폴더 복사를 요구받지 않는다.
- 업로드 실패는 토스트만이 아니라 모달 안에서 파일 단위로 확인된다.

### P0-4. 검색 기본 화면에서 내부 용어 숨기기

목표:
검색을 알고리즘 설명이 아니라 회의 업무 탐색으로 보이게 한다.

변경:

- `SPEAKER_00` placeholder → "화자 이름 또는 화자 1"
- "벡터 검색과 키워드 검색" → "의미가 비슷한 표현과 정확한 키워드를 함께 찾습니다"
- 결과 통계 기본값: "관련 결과 N건"
- score/source/vector/FTS는 "검색 세부 정보" 토글 안으로 이동
- 회의가 0건이면 추천 검색어 대신 녹음/가져오기 CTA 표시

완료 조건:

- 기본 검색 화면에 RAG, FTS, vector, score, SPEAKER_00가 보이지 않는다.
- 고급 세부 정보 토글을 켜면 기존 디버깅 정보는 유지된다.

### P0-5. 채팅 신뢰 가드

목표:
회의 근거 없는 답변이 제품 신뢰를 해치지 않게 한다.

변경:

- 회의/인덱스가 없으면 채팅 입력을 비활성화하고 "회의를 먼저 추가하세요" CTA를 제공한다.
- 검색 결과가 없을 때 LLM 일반 지식 답변으로 넘어가지 않는다.
- 답변은 기본적으로 citation을 포함한다.
- 출처 없는 경우에는 "회의 근거를 찾지 못했습니다" 상태로 종료한다.
- 전체 회의/현재 회의 범위를 화면 상단에서 더 명확히 표시한다.

완료 조건:

- 빈 데이터 상태에서 추천 질문이 보이지 않는다.
- `has_context=false`인 정상 답변은 생성되지 않거나, 사용자에게 명확히 "회의 근거 없음"으로 표시된다.
- 회의 근거가 있는 답변은 최소 1개 이상의 reference/citation을 표시한다.

### P0-6. Decision Wiki를 1급 업무 화면으로 승격

목표:
`/app/wiki`를 일반 위키가 아니라 결정사항 업무 화면으로 만든다.

변경:

- 탭/화면명을 "위키"보다 "결정사항" 중심으로 재정렬한다.
- 기본 화면에 최근 결정, 열린 액션, 프로젝트 현황을 보여준다.
- Decision List를 1급 컴포넌트로 만든다.
- Decision Detail에는 결정 내용, 배경, 후속 액션, 관련자, citation, 원문 이동을 둔다.
- Wiki compile/router가 비활성 또는 dry-run이면 활성화/백필 상태를 업무 언어로 안내한다.

완료 조건:

- 사용자가 "지난 결정"을 찾기 위해 채팅부터 열 필요가 없다.
- 각 결정은 원문 citation으로 이동할 수 있다.
- Wiki가 비어 있을 때 "회의 처리", "기존 회의 백필", "첫 회의 녹음" CTA 중 최소 하나를 제공한다.

### P0-7. 파이프라인 진행 상태를 산출물 기준으로 재정의

목표:
사용자가 "회의록은 완료됐는데 검색/채팅은 왜 안 되지?"를 겪지 않게 한다.

변경:

- 현재 6단계 진행 UI를 실제 산출물 기준으로 바꾼다.
- 권장 표시: 오디오 준비 → 전사 중 → 화자 구분 중 → 요약 중 → 검색 준비 중 → 결정사항 반영 중 → 완료
- `CHUNK/EMBED/WIKI_COMPILE`은 내부 단계명 대신 검색/결정사항 준비 상태로 보여준다.
- 실패 시 현재 단계, 영향 범위, 재시도 버튼을 복구 카드로 표시한다.

완료 조건:

- 검색 인덱스 실패가 발생해도 전사/요약 완료와 검색 미준비 상태를 분리해서 보여준다.
- 실패 상태는 토스트만으로 사라지지 않는다.

### P0-8. 설정 기본/문제해결/고급 분리

목표:
첫 사용자에게 설정 화면이 운영 콘솔처럼 보이지 않게 한다.

새 구조:

- 기본: 언어, 저장 위치, 테마, 오디오 입력
- 모델: 추천 STT/LLM 상태, 다른 모델 보기
- 문제 해결: STT 수동 다운로드, 검색 인덱스 복구, Wiki 백필, 로그/상태
- 고급: LLM backend, temperature, LLM 스킵, 환각 필터, A/B 테스트, custom model

완료 조건:

- 일반 탭 첫 화면에 backend, temperature, LLM 스킵, 환각 필터 세부값이 동시에 보이지 않는다.
- 추천 기본값을 유지하는 사용자는 모델 세부 선택 없이 첫 회의를 처리할 수 있다.

## 6. P1: 베타 안정화 계획

### P1-1. Shell layout contract 정리

- 글로벌 홈 버튼, 테마 버튼, 리소스 바, bulk action bar를 하나의 app toolbar/layout slot으로 정리한다.
- fixed/sticky 보정값 의존을 줄인다.
- resource monitor는 기본 사용자용 "상태 정상/주의"와 고급용 세부 리소스로 나눈다.

### P1-2. 디자인 시스템 계약 동기화

- `docs/design.md`와 `ui/web/tokens.css`의 spacing/radius/motion 토큰을 맞춘다.
- 토큰 이름을 유지할지 문서를 실제 코드에 맞출지 한쪽으로 결정한다.

## 8. 실행 순서

1. 카피 drift와 기본 UI 용어 수정
   가장 작고 위험이 낮다. 업로드, 검색, 홈 카피부터 정리한다.

2. 빈 데이터 상태와 채팅 신뢰 가드
   빈 회의 상태에서 검색/채팅/Wiki가 잘못된 기대를 만들지 않게 한다.

3. 홈 상태 적응형 구조
   첫 실행, 회의 0건, 처리 중, 회의 있음, 문제 있음 상태를 분리한다.

4. 파이프라인 진행/복구 카드
   실제 단계와 사용자 산출물 상태를 연결한다.

5. 설정 IA 분리
   기본/모델/문제해결/고급 구조로 옮긴다.

6. Decision Wiki 화면 재편
   Decision List/Detail/citation 중심으로 큰 변경을 진행한다.

7. Shell/layout/tokens 정리
   화면 구조 변경이 안정된 뒤 디자인 시스템 debt를 줄인다.

## 9. 측정 지표

- 첫 회의 완료율

- 첫 실행 체크리스트 통과율

- Decision extraction precision/recall
- citation validity
- Wiki search recall@5
- 출처 없는 채팅 답변 비율
- 설정 저장 없이 이탈하는 비율

품질 골든셋:

- 실제 한국어 회의 10~20개
- 각 회의의 수동 결정사항/액션/citation 라벨

- "전사 정확도는 낮지만 결정 추출은 맞는 경우"와 "전사는 맞지만 결정 추출이 틀린 경우"를 별도 기록

## 10. 구현 리스크와 완화

1. Wiki를 너무 빨리 전면화하는 리스크
   Wiki compile이 dry-run 또는 비활성인 상태에서는 "결정사항" 화면이 빈 약속처럼 보일 수 있다. 먼저 상태/백필/활성화 안내를 만든 뒤 Decision List를 전면화한다.

2. 고급 기능을 숨기다가 파워 유저를 잃는 리스크
   숨기는 것이 아니라 "고급/진단"으로 이동한다. 기존 기능 삭제는 하지 않는다.

3. 채팅 답변 제한으로 사용성이 줄어드는 리스크
   회의 근거가 없는 답변은 제품 신뢰를 해친다. 대신 검색 결과 없음 상태에서 "회의 추가", "검색어 바꾸기", "범위 변경"을 빠르게 제공한다.

4. 설정 IA 변경의 회귀 리스크
   API와 저장 방식은 유지하고, 우선 UI 섹션 재배치부터 한다. 저장 로직 변경은 별도 테스트 범위로 둔다.

5. Decision Wiki 품질 측정 부재
   UI 전면화 전에 citation validity와 extraction precision 기준을 만든다. 품질 기준 없이 화면만 키우면 신뢰를 잃는다.

## 11. 최종 합의문

Recap의 다음 UI/UX 개선은 예쁜 화면 만들기가 아니라 제품 계약 정리다.

사용자는 AI 파이프라인을 운영하려는 것이 아니라, 회의 후 "무엇이 결정됐고, 누가 무엇을 해야 하며, 그 근거가 어디 있는지"를 빠르게 확인하려고 한다.

따라서 P0는 첫 실행 준비, 첫 회의 생성, 검색/채팅 신뢰 가드, Decision Wiki의 1급 업무화, 실패 복구에 집중한다. 모델, 인덱스, 백필, A/B, 라우터, 리소스 세부 정보는 유지하되 기본 UI에서는 업무 언어 뒤로 접는다.

릴리즈 판단 기준은 다음 하나로 요약한다.

첫 성공 회의 후 사용자가 전사문, 요약, 결정사항, 액션, 프로젝트 현황, 원문 citation을 별도 설명 없이 확인할 수 있는가.
